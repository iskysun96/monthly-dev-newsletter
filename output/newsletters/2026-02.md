# Aptos Developer Monthly — 2026-02

> Your monthly digest of Aptos ecosystem updates

The February 2026 Aptos developer newsletter covers critical framework restructuring and protocol updates requiring immediate developer attention. The `aptos-framework` crate has been split into `aptos-framework-natives` and `aptos-release-bundle` packages to improve build performance, requiring dependency updates in existing projects. Additionally, significant VM security hardening, storage optimizations, and consensus improvements have been implemented alongside major SDK updates across TypeScript (v6.1.0), Rust (v0.4.0), and Go (v2.0.0) platforms.

## Breaking Changes

The Aptos framework has undergone a significant restructuring that affects build dependencies and package organization. This change addresses rebuild performance issues by separating runtime components from build tooling.

**Framework Crate Restructuring**

• **What broke**: The `aptos-framework` crate has been split into `aptos-framework-natives` and `aptos-release-bundle` packages
• **Why it changed**: The monolithic `aptos-framework` crate included both Move compiler tooling and runtime native functions, causing unnecessary rebuilds whenever compiler components changed
• **Migration required**: 
  - Update dependencies in `Cargo.toml` to reference `aptos-framework-natives` for runtime native functions
  - Use `aptos-release-bundle` for framework release artifacts
  - Review import statements and update module paths that previously referenced `aptos-framework` directly

**Build Performance Impact**

• **What broke**: Existing build scripts and CI pipelines may experience different rebuild patterns
• **Why it changed**: Breaking the dependency chain between compiler tooling and runtime components reduces cascading rebuilds
• **Migration required**:
  - Audit build dependencies to ensure you're importing the minimal required packages
  - Update CI caching strategies to account for the new package structure
  - Test build times to verify performance improvements in your development workflow

This restructuring primarily affects developers who directly depend on Aptos framework internals or have custom build processes that reference framework components.

## Protocol Updates

The Aptos protocol has undergone significant updates across virtual machine security, storage optimization, networking improvements, and consensus enhancements. Several changes require immediate attention from validators and node operators.

**Critical VM Security and Validation Updates**

The VM received multiple security hardening changes that affect bytecode validation and execution. Bounds checking and recursion handling were fixed, with changes to production bytecode verification limits for struct, function, and script bounds behind timed feature flags. This may cause previously-valid modules or scripts to be rejected. Type depth tracking and async runtime checks were improved to prevent execution-replay divergence. Stack size validation was added for struct unpacking and closure calls to catch malformed bytecode that could cause stack corruption. The VM now uses full transaction size for gas calculations and checks closure depth bounds at construction time.

**Storage Layer Optimizations**

Storage underwent major performance and reliability improvements. RocksDB ticker statistics are now exposed via Prometheus for better monitoring. A critical race condition in HotState was fixed using RCU/deferred merge patterns. The pruner now sets `fill_cache(false)` on RocksDB reads to prevent evicting hot data from the block cache. StateKvDb was refactored to follow the StateMerkleDb pattern with separate hot/cold instances. Storage sharding is now always enabled, with non-sharding code paths removed. A background thread was added to clean up stale JMT nodes from a bug fixed last October - nodes bootstrapped before October may see temporary disk usage increases.

**Networking and State Sync Improvements**

The networking stack was streamlined by replacing the network health checker with the peer monitoring service for disconnect logic. Legacy subscription logic was removed from PeersAndMetadata, and old netbench code was cleaned up. State sync performance improved significantly by consolidating four separate tokio runtimes into one shared runtime, reducing thread count from 176 to a more reasonable number on multi-core systems.

**Consensus and Randomness Enhancements**

Consensus received optimizations for randomness handling and payload management. Expensive cryptographic aggregation is now deferred for blocks without randomness-consuming transactions. Optimistic randomness share verification was added, replacing per-share verification with batch verification at aggregation time. The proof manager now exclusively uses OptQuorumStore payload variants, with legacy payload types deprecated. Initial support for encrypted transaction decryption and DKG integration was added, including ChunkyDKGManager integration with the epoch manager.

**Move Language and Compiler Updates**

The Move compiler gained struct and enum visibility modifiers in a multi-step rollout, with bytecode verification support added. Format string validation was implemented for assert macros. Unused private functions, structs, and constants now generate warnings. Signed integer return parsing was fixed in compiler v2. The Move Prover received behavioral predicates support for verifying higher-order functions using closures.

**Developer Tooling**

A new `aptos-move-flow` crate was introduced providing AI tool integration for Move development, including plugin generation and MCP stdio server capabilities. Legacy Move IR (.mvir) tests are being migrated to Move assembly (.masm) format as part of removing the deprecated Move IR compiler.

**Node Operator Impact**

Validators and fullnode operators should upgrade to the latest releases, particularly the hotfix versions addressing critical VM security issues. Nodes may experience temporary disk usage increases during stale data cleanup. The networking and storage optimizations should improve overall node performance and resource utilization.

## SDK & Tooling

**Aptos TypeScript SDK** version 6.1.0 is in preparation with new features, changes, and bug fixes. The SDK now includes HTTP/2 compatibility checks for Bun environments and will throw errors if HTTP/2 isn't explicitly disabled. A gas estimation regression has been fixed with simulation retry logic when servers return `MAX_GAS_UNITS_BELOW_MIN_TRANSACTION_GAS_UNITS`. Security vulnerabilities in dependencies have been resolved through `pnpm.overrides`. New CI workflows validate compatibility across Bun, Deno, and web environments. GitHub CI actions have been updated to latest versions.

**Aptos Rust SDK** version 0.4.0 includes comprehensive security audit remediations and ECDSA signature hardening. The SDK has migrated from the `hex` crate to `const-hex` for improved performance and const compatibility. Multi-agent fee-payer simulation changes have been reverted. Circular dependency issues for releases have been fixed. New CI jobs support Linux ARM, macOS x86, and Windows builds. Documentation publishing to docs.rs has been configured. E2e tests now properly run in CI with the `-- --ignored` flag. Dependencies have been updated including `reqwest` to v0.13 and `keccak` to 0.1.6.

**Aptos Go SDK** v2.0.0 represents a complete rewrite with more idiomatic Go features. Version 1.12.0 adds support for signed integers and includes type conversion improvements. Both v1 and v2 branches received security updates with `filippo.io/edwards25519` bumped from 1.1.0 to 1.1.1. Documentation has been updated with godoc URLs.

**Aptos Wallet Adapter** core version 8.4.0 adds cross-chain wallets to the registry including Coinbase Wallet and Watchee wallet. React adapter 8.2.1 fixes async `autoConnect` and late wallet injection issues. Cross-chain core 5.8.0 adds Aptos Gas Station support for CCTP withdraw transactions and fixes Solana multi-signer transaction ordering for Phantom wallet compatibility. Server-side Solana claim support has been added for seamless CCTP withdrawals.

**Forklift** version 1.2.0 adds Aptos CLI version validation at `Harness` construction time, replacing cryptic errors with actionable messages. The `MIN_CLI_VERSION` constant and `compareVersions` helper are now exported. New features include `newBlock`, `advanceEpoch`, and `profileGas` functionality. CI has been consolidated from 3 test workflows into a single matrix workflow with Aptos CLI caching and concurrency controls. TypeScript has been bumped to ^5.9.3 across packages.

**Aptos Explorer** has migrated from ESLint and Prettier to Biome for improved performance. Security vulnerabilities have been resolved including Rollup arbitrary file write and Minimatch ReDoS issues. The explorer now includes SEO improvements with aria-labels, SSR head metadata, and LLM discoverability features. Token verification status visibility has been enhanced with new verification banners. Local device detection now only activates when users explicitly select "localnet".

**Aptos Indexer Processor SDK** version 169 adds backup gRPC endpoint failover support with configurable endpoints and independent auth tokens.

**Migration Notes**: 
• Rust SDK users should update dependency from `hex` to `const-hex` and use re-exported `const_hex` from `aptos-sdk`
• TypeScript SDK users running in Bun must explicitly disable HTTP/2 with `clientConfig: { http2: false }`
• Wallet adapter users can now access cross-chain wallets through the updated registry
• Forklift users should verify minimum CLI version requirements with the new validation system

## New Features

The Aptos ecosystem has introduced a new **Aggregated type for PVSS (Publicly Verifiable Secret Sharing) aggregation** in the aptos-dkg module. This enhancement addresses performance and code complexity issues in the distributed key generation system.

The new Aggregated type enables developers working with DKG implementations to:

• **Improved transcript handling**: The chunky system now stores affine elements instead of projective elements in transcripts, significantly simplifying code structure and improving performance across various operations

• **Optimized aggregation operations**: Resolves inefficiencies in the `aggregate_with()` method that previously caused performance bottlenecks when working with naive implementations

• **Enhanced DKG reliability**: Provides a more robust foundation for applications requiring distributed key generation, such as threshold cryptography schemes and multi-party computation protocols

This change primarily impacts developers building:
- Consensus mechanisms requiring distributed key generation
- Multi-signature schemes leveraging PVSS
- Applications implementing threshold cryptography
- Custom validator or node software requiring secure key sharing

The modification maintains backward compatibility while providing performance improvements for existing DKG-dependent applications. Developers currently using the aptos-dkg module should review their aggregation logic to take advantage of the optimized Aggregated type, particularly if they're implementing custom transcript handling or working directly with PVSS operations.

Full implementation details and breaking changes are available in the [pull request](https://github.com/aptos-labs/aptos-core/pull/18514).

## Community Highlights

The Aptos ecosystem saw significant infrastructure improvements and feature additions this period. Major performance optimizations include jemalloc tuning with per-CPU arenas and 2MB huge pages, thread count reductions across various services to address oversubscription issues, and Block-STM pre-write optimizations for MVHashMap contention reduction.

Key development experience enhancements were delivered through several initiatives:

• **CLI and tooling improvements**: The Move CLI was extracted into a standalone `aptos-move-cli` crate for independent building and testing. Gas profiler received a major UI overhaul with interactive reports, sortable tables, and execution trace views.

• **Testing and debugging capabilities**: Transaction Simulation Sessions (TSS) gained `new_block` and `advance_epoch` functionality plus gas profiling support. Signed integer support was added to transactional test arguments, and all aptos-node threads were named for better debugging.

• **Code cleanup and modernization**: Legacy in-node indexer code was removed in favor of separate transaction streaming services. The deprecated node-checker tool was entirely deleted, and move-errmapgen was removed as error map generation moved to the framework.

Encrypted transaction support progressed significantly with end-to-end plumbing from core types through DKG key derivation, Move framework integration, and VM support. A comprehensive smoke test validates the full encrypted transaction lifecycle across 4-validator swarms.

Cryptographic infrastructure saw substantial updates including Shplonked PCS implementation, PVSS trait redesigns, and chunky DKG performance improvements through custom batch multiplication and affine element usage. The randomness fast path was disabled by default while maintaining standard randomness functionality.

Framework and VM enhancements include chunky DKG reconfiguration support, emergency liveness preservation for empty validator sets, enum variant descriptions in node API ABIs, and improved Move specification formatting in the sourcifier.

Docker infrastructure was modernized with migration from Debian Bullseye to Trixie, parallel build splitting across 4 builders, and Rust toolchain updates to version 1.91. The faucet migrated to orderless transactions using nonce-based replay protection instead of sequence numbers.
