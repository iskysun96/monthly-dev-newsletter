Breaking Changes

This PR extracts native functions and release bundles from the main `aptos-framework` crate into separate crates (`aptos-framework-natives` and `aptos-release-bundle`) to optimize build times by breaking dependency chains.

**What broke:**
• Direct imports from `aptos-framework` for native function implementations may fail
• Build scripts or tooling that relied on the monolithic `aptos-framework` structure could break
• Custom native function integrations may need updated import paths

**Why it changed:**
• The `aptos-framework` crate previously bundled both build tooling and runtime natives, causing unnecessary rebuilds when compiler components changed
• Separating concerns reduces build times and creates cleaner dependency boundaries

**Migration actions:**
• Update import statements to use `aptos-framework-natives` for native function implementations
• Modify build dependencies to reference the appropriate split crate (`aptos-framework-natives` or `aptos-release-bundle`) instead of the monolithic `aptos-framework`
• Review custom tooling that depends on `aptos-framework` internal structure and update to use the new crate organization
• Test builds to ensure all native function references resolve correctly with the new crate structure

**Explorer** enhanced token verification visibility with new `VerificationBanner` component showing prominent success banners for verified tokens and warnings for unverified ones. Fixed FA transfer balance display for Transfer, Mint, Burn, and Frozen event types. Added responsive layout improvements for transaction action display. Protocol Updates

The latest Aptos protocol updates include several significant changes across VM security, storage optimization, consensus improvements, and developer tooling.

**VM Security and Type System Enhancements**

The VM received critical security improvements with better bounds checking and recursion handling, including fixes for bytecode verification limits behind timed feature flags. Stack size validation was enhanced for struct unpacking and closure calls to prevent stack corruption. Type depth tracking during gas charging was improved, and async runtime type checks were strengthened to prevent execution-replay divergence. Struct layout caching now uses `Arc` for better memory efficiency during layout construction.

**Storage Layer Optimizations**

Storage performance saw major improvements through several changes:
• RocksDB ticker statistics are now exposed via Prometheus for better observability
• Pruner operations now use `fill_cache(false)` to prevent evicting hot data from the block cache
• State-sync was optimized by consolidating four separate tokio runtimes into one shared runtime, reducing thread count from 176 to a more reasonable number on multi-core systems
• A race condition in `HotState` was fixed using RCU/deferred merge patterns
• Storage sharding is now always enabled, with non-sharding code paths removed
• `StateKvDb` was refactored to follow the `StateMerkleDb` pattern with separate hot/cold instances

**Consensus and Randomness Improvements**

Consensus received performance optimizations for randomness handling:
• Expensive cryptographic aggregation is now deferred for blocks without randomness-consuming transactions
• Optimistic randomness share verification reduces per-share overhead by batch-verifying at aggregation time
• Legacy payload enum variants were deprecated in favor of `OptQuorumStore` exclusively
• New chunky DKG (Distributed Key Generation) infrastructure was added with `ChunkyDKGManager` integration

**Network Stack Cleanup**

The networking layer underwent significant refactoring:
• The network health checker was replaced with the peer monitoring service for disconnect logic
• Subscription logic was removed from `PeersAndMetadata` to reduce technical debt
• Legacy netbench code was removed entirely

**Developer Experience Improvements**

Move language development received several enhancements:
• Struct and enum visibility modifiers were added with bytecode verification support
• Compiler v2 now enables warnings for unused private functions, structs, and constants
• Format string validation was added for assert macros
• Signed integer return parsing was fixed
• A new `aptos-move-flow` crate provides AI tool integration for Move development
• Move Prover gained support for behavioral predicates and function values (closures)

**Breaking Changes and Migration Notes**

Validators and node operators should note that some changes are behind timed feature flags and may affect previously-valid modules. The storage cleanup may cause temporary disk usage increases for nodes bootstrapped before October. Several dead storage APIs were removed, which may affect custom integrations.

These updates focus heavily on performance optimization, security hardening, and developer experience improvements while maintaining backward compatibility through feature flags where necessary.

New Features

The Aptos ecosystem has introduced a new Aggregated type for PVSS (Publicly Verifiable Secret Sharing) aggregation within the aptos-dkg module. This enhancement improves the efficiency of distributed key generation operations by optimizing how transcript data is stored and processed.

For developers working with cryptographic applications or building systems that require distributed key generation, this change provides:

• More efficient aggregation operations through the use of affine elements instead of projective elements in transcripts
• Simplified code paths for PVSS-related functionality
• Improved performance characteristics for applications utilizing the DKG (Distributed Key Generation) system

This update primarily benefits developers building advanced cryptographic applications, multi-party computation systems, or consensus mechanisms that rely on distributed key generation. The optimization reduces computational overhead while maintaining the security guarantees of the PVSS scheme.

The change is part of ongoing improvements to Aptos's cryptographic infrastructure, specifically targeting the "chunky" transcript modifications that will streamline various operations throughout the DKG system.

Community Highlights

The Aptos ecosystem has seen significant developer tooling improvements and new cryptographic features. Forge now includes Prometheus metrics for cluster spin-up and transaction emitter monitoring, enhancing observability for testing infrastructure. The node API has been enhanced to include enum variants in ABI responses, making it easier for developers to understand Move module interfaces.

Several major developer experience improvements have landed:

• **Transaction Simulation Sessions (TSS)** now support gas profiling with HTML reports and can simulate block/epoch boundaries through `new_block` and `advance_epoch` methods, enabling more comprehensive testing scenarios.

• **Transactional test arguments** now accept signed integers (e.g., `-1i64`), removing previous parsing limitations with negative values.

• **Move assembler improvements** include native function support for better roundtripping and elimination of .mvir files in favor of .masm format across the codebase.

The cryptographic infrastructure has been significantly expanded with **encrypted transaction support** now available end-to-end, including API endpoints, SDK builders, VM integration, and execution layer support. A new **Chunky DKG** implementation provides an alternative distributed key generation approach with Move framework integration and reconfiguration support.

Major cleanup efforts have streamlined the codebase by removing deprecated components like the legacy in-node indexer, node-checker tool, and move-cli crate. The move-docgen and move-abigen tools have been relocated from the prover directory to a more appropriate tools directory, reducing unnecessary dependencies.

Performance optimizations include jemalloc tuning with per-CPU arenas and huge page support, thread count reductions for various services, and Block-STM pre-write optimizations for frequently accessed resources. The Docker infrastructure has been modernized with a migration from Debian Bullseye to Trixie and split into four parallel builders for faster builds.

The Rust toolchain has been updated to version 1.91, and the faucet system now uses orderless transactions with nonce-based replay protection instead of sequence numbers. Documentation has been enhanced to be LLM-ready, improving accessibility for AI-assisted development workflows.
